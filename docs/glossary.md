# Glossaire — Ubiquitous Language

> Termes du bounded context **CodeImpact**, utilisés dans le code, les docs, et les conversations.

| Terme | Nature | Définition |
|---|---|---|
| AnalysisTarget | VO | Fichier ou projet soumis à l'analyse. Porte un `path` et un `TargetType` (File / Project). |
| CodeLocation | VO | Position précise dans le code: `file_path`, `line`, `column`. Validée au constructeur (line >= 1, column >= 1). |
| CodeMetrics | VO | Résultat de l'analyse proactive: complexité cyclomatique, complexité transitive, complexité cachée (somme additive des `hidden` par fonction — voir [[ADR-0012]]), profondeur d'appel, warnings, impact économique. Porte aussi `capabilities: Option<LanguageCapabilities>` (US14-T3, #33) : la capacité par-métrique par-langage déclarée par le parseur, qui **circule jusqu'aux writers** pour piloter le rendu `n/a` (voir [[ADR-0021]]). `None` = tout supporté (cas Rust, sortie byte-inchangée). |
| FunctionDetail | VO | Métriques d'**une** fonction. Champs **privés**, construit par `new()`. Stocke `direct` et `hidden` ; **dérive** `transitive() = direct + hidden`. L'état illégal (`transitive < direct`) est donc **inconstructible**, y compris depuis un futur adaptateur FFI — impossibilité structurelle plutôt que garde-fou runtime. Voir [[ADR-0012]]. |
| EconomicImpact | VO | Coût estimé: CPU (μ$), mémoire (bytes), coût total (μ$), niveau (low/moderate/high/critical). |
| EconomicImpactEstimator | Domain Service | Calcule EconomicImpact à partir de CodeMetrics, ParsedFunction, et CallGraph via formules heuristiques. |
| MicroDollars | Unité | 1 μ$ = 10⁻⁶ $. Unité de coût CPU basée sur le pricing cloud (~$0.10/CPU-heure). 1 μ$ ≈ 10⁷ cycles CPU. |
| EcologicalImpact | VO | Impact CO2 (g) et énergie (J) dérivé de l'impact économique via un facteur régional (gCO2/kWh). |
| EfficiencyClass | Enum | Label A–G basé sur le CO2 (A = très faible, G = extrême). |
| AnalysisRule | Enum | Règle d'analyse proactive: CyclomaticComplexity, IoInLoops, NestedDepth, AllocationHotspots. |
| AnalysisReport | DTO | Regroupe les métriques, impacts, et recommandations. Sortie du use case. |
| ProactiveAnalyzer | Domain Service | Analyse statique du code source (AST, patterns). Orchestre l'estimation d'impact économique. |
| ReactiveAnalyzer | Domain Service | Analyse dynamique via exécution instrumentée des tests. |
| StressTestRun | VO | Résultat d'un run de tests instrumenté: durée, CPU, mémoire (ces deux-là en `Measurement`), tests passés/total, filtre. Sur un workspace, c'est l'**agrégat** des N binaires de test (voir Agrégation). |
| Measurement | VO | `Available(T)` ou `Unmeasurable(raison)`. Type somme qui rend **structurellement impossible** de confondre « mesuré à 0 » et « pas su mesurer » — un `Available(0)` honnête reste légitime (voir [[ADR-0010]]). |
| UnmeasurableReason | Enum | Pourquoi une mesure manque. `NoSampler`: pas de sondeur (`/usr/bin/time`) sur l'hôte. `NoTestsExecuted`: aucun test n'a tourné — un run qui n'a rien exercé n'a aucun coût honnête à rapporter, si bien échantillonné que soit son processus (voir [[ADR-0011]]). |
| Agrégation | Loi de domaine | Pliage des N `StressTestRun` d'un workspace en un seul (`StressTestRun::aggregate`). Durée et tests: **somme**. CPU: somme, mais `Unmeasurable` si **un seul** run l'est. Mémoire: **max** — un pic RSS de processus qui ne coexistent jamais n'est pas une somme. Voir [[ADR-0011]]. |
| Ligne de résumé | Concept | La ligne `test result:` qu'un binaire libtest imprime **toujours** s'il va au bout — succès, échec, ou zéro test. Son **absence** est le discriminant d'un binaire **crashé**. Le code de sortie ne l'est pas: un binaire dont des tests échouent sort en non-zéro sur son chemin **nominal**, et les tests rouges doivent rester mesurables (voir [[ADR-0011]]). |
| DependencyContext | VO | Contexte **neutre (agnostique du langage)** dont un `CodeParser` a besoin pour résoudre les dépendances déclarées d'un fichier vers des fichiers réels : `{ current_file, project_root, available_files }`. Ne porte **aucune** opinion sur ce qu'est un « module » — la sémantique `crate::`/`super::`/`.rs`/`mod.rs` est **privée à l'adaptateur** Rust. Consommé par le use case `RunAnalysis`. Introduit par US14-T1 (#32). Voir [[ADR-0018]]. |
| resolve_dependencies | Méthode de port (CodeParser) | `resolve_dependencies(source, &DependencyContext) -> Vec<PathBuf>` : rend les dépendances déclarées d'un fichier **déjà résolues en chemins réels** contre `available_files`. Remplace l'ancien `parse_file_dependencies -> Vec<String>` et son protocole *stringly-typed* `"mod:"`/`"use:"` — **ces jetons ont disparu, plus rien hors de l'adaptateur ne les connaît**. Une dépendance non résolvable est **absente** du résultat, jamais une erreur. Le domaine ne voit **que** des `PathBuf`, jamais la syntaxe d'un langage. Voir [[ADR-0018]], [[ADR-0013]]. |
| source files (`list_source_files`) | Concept / Méthode de port (CodeReader) | Le terme du domaine est **« fichiers source »**, pas « fichiers Rust ». `list_source_files(dir, extensions: &[&str])` filtre sur l'ensemble d'extensions **passé par la racine de composition** (`&["rs"]` aujourd'hui ; `&["cs"]`, `&["ts","tsx"]` pour un futur adaptateur), au lieu d'un `ext == "rs"` codé en dur. Le stub de test suit : `source_files` / `add_source_file` (ex-`rust_files` / `add_rust_file`). Voir [[ADR-0018]]. |
| Branchement (`branch_arms` / `LargeBranching`) | Concept | Nombre de **branches** d'une structure de sélection, et le warning qui le sanctionne au-delà de `max_branch_arms`. Le domaine dit **branchement**, pas `match` : `match` est un mot-clé **Rust** (l'adaptateur en parle), le branchement est le **concept métier** que tout langage partage (`switch` en C#, `switch`/ternaire en TS). Renommage US14-T1 : `match_arms`→`branch_arms`, `LargeMatch`→`LargeBranching`, `max_match_arms`→`max_branch_arms`. Même discipline de vocabulaire qu'[[ADR-0013]]. Voir [[ADR-0018]]. |
| CodeReaderPort | Port | Interface pour lire le code source depuis le filesystem : `read_source(target)` + `list_source_files(dir, extensions)` (agnostique du langage, voir [[ADR-0018]]). |
| ProfilerPort | Port | Interface pour mesurer l'impact réel (CPU/mem/IO). L'implémentation P0 utilise des heuristiques. |
| TestRunnerPort | Port | Interface pour exécuter les tests avec instrumentation. |
| ReportWriterPort | Port | Interface pour produire le rapport (console, JSON, HTML). `write_json` / `write_html` / `write_project_json` retournent le document en `String` ; **l'écriture fichier vit dans les primaries** (`write_report_file`, partagée par JSON et HTML). `write_console` écrit **directement sur un flux** et ne retourne rien — d'où le refus explicite de `--format console -o` (voir ci-dessous). |
| `-o` (option de sortie) | Concept | **Aucun format n'ignore `-o` en silence.** JSON et HTML **écrivent le fichier** ; console **refuse explicitement** (exit 1, message clair), parce que `write_console` écrit sur un flux sans retourner de `String` et que l'y brancher changerait le contrat du port. C'est la lecture retenue de « cohérent entre les formats » : le mal nommé par le défaut #48 était **le silence**, pas l'asymétrie. Un `-o` ignoré sans un mot est une **perte de données sans avertissement**. |
| HtmlReport | Concept | Rapport visuel self-contained: un seul `.html` (CSS/JS/data JSON inline, fonts système, zéro asset externe) ouvrable en `file://`. Produit par `HtmlReportWriter`. Vue projet (liste fichiers) en T1; vue node détail en T2. |
| ImpactScore | Concept | Score d'impact affiché dans le rapport HTML = `transitive_complexity()`. Heuristique de présentation, PAS une métrique domaine (voir [[html-report]] ADR-8.8). Barres normalisées au max du projet. |
| DataIsland | Concept | Bloc `<script id="ci-data" type="application/json">` portant le view-model sérialisé. Échappé via `json_island_escape` (breakout `</script>`); rendu côté client par `textContent`/`createElement` (défense XSS structurelle). |
| CallGraph | VO | Graphe d'appels entre fonctions, dérivé du parsing AST. Utilisé pour la complexité transitive et la profondeur. |
| CyclomaticComplexity | Concept | Nombre de chemins linéairement indépendants dans le code. Mesure statique de complexité structurelle. |
| TransitiveComplexity | Concept | Somme des complexités de toutes les fonctions appelées (directement ou indirectement). |
| HiddenComplexity | Concept | Complexité atteignable **à travers les appels** d'une fonction : `hidden(f) = Σ direct(g)` pour chaque `g` du **sous-graphe atteignable** depuis `f`, chaque fonction distincte comptée **une seule fois** (lire `g` deux fois n'est pas deux fois le travail). Toujours ≥ 0 par construction. **C'est une propriété de la FONCTION**, qui s'agrège **additivement** : `hidden(fichier) = Σ_f hidden(f)`, `hidden(projet) = Σ_fichiers hidden(fichier)`. Elle ne se calcule **jamais** en soustrayant deux agrégats (`ΣT − ΣC`) : `C` (fichier) et `T` (fonction) ne sont pas dans la même unité, et la soustraction exige un clamp qui masque les violations d'invariant. Voir [[ADR-0012]]. |
| TransitiveComplexity | Concept | Coût de **compréhension** d'une fonction : `transitive(f) = direct(f) + hidden(f)`. Dérivée, jamais stockée — d'où `transitive ≥ direct` comme vérité arithmétique, sans garde-fou runtime. Bornée par la somme des complexités directes du fichier, donc non débordable par construction. Ce n'est **pas** une somme sur les chemins d'exécution (c'était le défaut corrigé par [[ADR-0012]]). |
| ProjectMetrics | VO | **Unique source de vérité** des nombres de niveau-rapport (fichiers, complexités directe/transitive/cachée, profondeur, cycles, warnings, critiques, I/O-en-boucle, hotspots, impacts). Calculée une seule fois par `FileConsumptionGraph::aggregated_metrics()`. Les writers console, JSON et HTML la **rendent** — aucun n'a le droit de recalculer un agrégat. Voir [[ADR-0012]]. |
| ComplexityWarning | VO | Avertissement de complexité **porteur d'une sévérité** (`Critical` / `Warning`). Compté par la tuile `Warnings`. |
| IoInLoopWarning | VO | I/O détectée dans une boucle. **N'a PAS de sévérité** — l'agréger à un décompte de « critiques » est un non-sens (cause du défaut #49). Classe distincte, tuile distincte. Voir [[ADR-0012]]. |
| LoopCall | VO | Un appel **niché dans une boucle** : `{ name, line, col, io }`. Le parseur enregistre **tout** appel à `loop_depth > 0` — c'est **un fait**, que chaque détecteur **interprète** : `io_in_loops_detector` filtre sur `io == Io`, `detect_quadratic_loops` les lit tous. Le champ s'appelait `calls_in_loops` et ne contenait que les I/O : **son nom mentait**, et c'était la cause du défaut #47. Le booléen `is_io` a cédé la place à [[ADR-0016]]'s `IoClassification` (#56) : un bool rendait l'abstention non représentable. Voir [[ADR-0013]]. |
| IoClassification | VO | Classification d'un `LoopCall` à **trois états** : `Io` (preuve de type — le receveur est résolu vers un type d'I/O connu), `NotIo` (type résolu non-I/O, type déclaré localement, ou nom non suspect sur receveur irrésolu), `Unknown` (**abstention** : receveur irrésolu ET nom de méthode suspect — l'outil ne dit ni « I/O » ni « pas d'I/O »). Remplace `is_io: bool` (US5, #56). Règle d'or : **le type affirme, le nom s'abstient** — un nom seul ne produit jamais un warning. `match` exhaustif obligatoire, aucun `_` catch-all : chaque consommateur décide explicitement quoi faire de l'incertitude. Transposition de la mécanique `Measurable`/`Unmeasurable` d'[[ADR-0010]] à la classification d'un appel. Voir [[ADR-0016]]. |
| Abstention | Concept | Ce que fait l'outil quand il **ne peut pas trancher** (US5) : compter, jamais accuser. Les `Unknown` remontent comme **compteur agrégé** (« N appels en boucle non classifiables », par fichier + total projet) sur les trois surfaces (JSON additif, console, HTML) — **jamais** en détail par ligne : une abstention transformée en pseudo-warning crierait au loup. Calibré sur deux corpus (ce dépôt + ripgrep) : 0 % de faux positifs `Io`, ≤ 1,4 % d'abstention. Voir [[ADR-0016]], [[ADR-0010]]. |
| QuadraticLoop | Concept | Motif O(n²) **probable** : une boucle dont un appel **niché** cible une fonction qui boucle elle-même. Une **récursion d'arbre** (la boucle itère les enfants, l'appel descend) est du O(n) et n'en est **pas** une — d'où l'exclusion des appelés cycliques. Un appel **séquentiel après** la boucle n'en est pas une non plus. Voir [[ADR-0013]]. |
| QualifiedName | Concept | Nom d'une fonction **qualifié par sa portée englobante** : `Type::method` pour une méthode d'`impl` (inhérent **ou** de trait), `Trait::method` pour une méthode par défaut de trait, `mod::Type::method` sous un `mod` inline, et le nom **nu** pour une `fn` libre. Le schéma est de la **syntaxe** — il vit dans l'adaptateur, l'hexagone ne voit qu'une `String` opaque. Le parseur **garantit l'unicité** des noms qu'il émet (suffixe `#2` sur collision) : un doublon écraserait silencieusement une fonction entière dans `CallGraph::build`. Rejeté : `<Type as Trait>::method` — il casserait la résolution des appels `self.foo()`, que `syn` ne peut pas inférer. Voir [[ADR-0014]]. |
| IntraTypeResolution | Concept | Ce que le graphe d'appel **sait** résoudre : `self.m()` et `Self::m()` dans un `impl Type` → `Type::m` (61/61, exact). Ce qu'il **refuse** de résoudre : `x.m()` et `self.field.m()` restent des identifiants **nus**. Résoudre par homonymie fabriquerait des arêtes vers du code jamais appelé (2275 appels, 8 homonymes) — **fabriquer une arête est le pendant exact du `0` confiant** d'[[ADR-0010]]. Le graphe est donc **intra-fichier**, et pour les méthodes **intra-type** : un appel externe pend, et contribue `0` — ce qui est *correct*, ce code n'a pas été mesuré. Voir [[ADR-0014]]. |
| NiveauNone | Concept | Troisième état de mesure, entre « mesuré » et « non mesuré » : le fichier a été **parfaitement lu**, il n'a simplement **rien à mesurer** (déclaration de trait sans corps, type de données pur, `mod.rs` de ré-export). Publié `complexity_level: "none"` — neutre en console, **gris** en HTML. **Jamais `low`, jamais vert** : « rien à signaler » n'est pas « propre ». À ne pas confondre avec un fichier **non mesuré**. Voir [[ADR-0014]]. |
| UnmeasurableFile | VO | Fichier que l'outil **a échoué à lire** — `{ path, reason }` avec `SourceUnparseable` (le parseur n'a pas su), `SourceUnreadable` (le disque n'a pas su), `SourceTooLarge` (source au-delà de la borne d'entrée, voir [[ADR-0006]]) ou `SourceTooComplex` (parsing pathologiquement imbriqué qui ferait déborder la pile — refusé pour **ce fichier seul** par la sonde sous-processus, jamais un crash du scan, voir [[ADR-0015]]). Il **apparaît** dans le rapport, marqué NON MESURÉ, et il est **exclu de toute somme** — il n'a aucun nombre à donner. Avant [[ADR-0014]] il n'était pas remonté à `0` : **il disparaissait purement du rapport**, et `total_files` le sous-comptait. Même mensonge qu'un `0` confiant, un étage plus haut. Voir [[ADR-0010]], [[ADR-0014]]. |
| AlertThresholds | VO | Seuils d'alerte que l'utilisateur fixe sur l'impact agrégé (US8, #8). Deux champs `Option<f64>` : `max_energy_kwh` et `max_co2_grams` — **énergie (kWh) et CO2 (g) uniquement**, jamais le coût CPU, la complexité ni l'I/O-en-boucle. L'énergie provient d'`EcologicalImpact::energy_joules()` convertie en kWh (`/ KWH_TO_JOULES`) à la frontière. Auto-valide au constructeur (`new` rejette NaN/Infinity/négatif ; `0` est un seuil valide, maximalement strict). Construit par les adaptateurs (flags CLI + fichier `.codeimpact.json`) puis fusionné par `from_sources` (le CLI l'emporte sur le fichier, par métrique). La comparaison vit **entièrement dans l'hexagone zéro-dep** (voir [[ADR-0017]], [[ADR-0001]]). |
| Seuil / Threshold | Concept | La limite au-delà de laquelle l'impact déclenche une alerte. Comparée à l'**agrégat niveau-projet** (`ProjectMetrics`) ; une cible fichier-seul compare l'impact de ce fichier. **Jamais par-fonction.** Règle d'or ADR-0010 transposée : `evaluate` ne compare que sur `(Some, Some)` — une énergie/CO2 `Unmeasurable`/absent (`None`) ne peut **jamais** produire un dépassement. Fabriquer un dépassement depuis une mesure manquante serait le pendant exact du `0` confiant. Voir [[ADR-0017]], [[ADR-0010]]. |
| ThresholdReport | VO | Résultat pur de `AlertThresholds::evaluate(cpu, co2)` : la liste des dépassements (`ThresholdBreach`) constatés, éventuellement vide. Immuable, **constructeur privé** — seul `evaluate` peut en fabriquer un. `has_breach()` lu par la couche de sortie ; `summary()` alimente le message `--strict`. Distinct de « aucun seuil configuré » (pas de report du tout) — la distinction est portée par un `Option<ThresholdReport>`, pas aplatie. |
| Dépassement / Breach | Concept | Un `ThresholdBreach` : une métrique (`BreachedMetric` = Energy ou Co2), sa limite, sa valeur constatée, et l'excès (`excess() > 0` par construction). Rendu **de façon identique** sur console / JSON / HTML / stderr `--strict` par un **unique** renderer partagé (`humanize::render_threshold_warning`) — aucune logique de formatage dupliquée, donc aucune dérive entre surfaces. |
| Mode strict (`--strict`) | Concept | Flag qui transforme un dépassement en **échec de processus** : sans `--strict`, un dépassement **avertit** et sort `0` ; avec, il sort **`3`** (distinct de `1` = erreur runtime/input, et du `2` réservé par clap aux erreurs d'arg). Le code de sortie est **décidé dans le domaine** (`GatedOutput<T>` porte le verdict), seulement **mappé** dans `main.rs` (`gated_exit_code` : `strict && has_breach → 3`) — zéro comparaison dans la racine de composition. Applicable à `analyze` **et** `stress-test`. Voir [[ADR-0017]]. |
| `.codeimpact.json` | Concept | Fichier de config projet, lu via `ConfigReaderPort` / `FileSystemConfigReader` (`serde_json` en secondaries — l'hexagone ne voit jamais serde, voir [[ADR-0017]] DIP). Rend un `AnalysisConfig` déjà validé. Depuis US15 (#31) le schéma **complet est déclaré** : `thresholds` (US8) + `include`/`exclude`/`respectGitignore` (US15) **consommés**, plus les clés forward-compat `languages`/`sourceRoots`/`extensions`/`parser`/`ioSignatures` **parsées mais inertes** (câblées par #33/#34 sans rouvrir le contrat). La tolérance aux clés inconnues d'[[ADR-0017]] laisse place à **`#[serde(deny_unknown_fields)]`** (une faute de frappe échoue au lieu d'être avalée — changement assumé, voir [[ADR-0019]]). Découvert via `--config <path>` ou auto-recherche (dossier cible puis cwd). Lecture disciplinée ADR-0006 (canonicalize parent seul, `symlink_metadata` refuse symlink/FIFO/dossier avant toute lecture, plafond de taille, aucune fuite de path). Voir [[ADR-0017]], [[ADR-0019]], [[ADR-0006]]. |
| AnalysisConfig | VO | Objet de config projet lu depuis `.codeimpact.json` (US15, #31) : **composition immuable** de deux VO déjà validés — `thresholds: AlertThresholds` (US8) et `filter: FileFilter` (US15). `defaults()` = aucun seuil + `FileFilter::unrestricted()` (fichier absent ⇒ comportement byte-identique au pré-US15, D4). **Pas un Aggregate DDD** : VO composite sans identité ni cycle de vie (le MVP reste sans Aggregate). Rendu par `ConfigReaderPort::read_config`. Voir [[ADR-0019]]. |
| FileFilter | VO | Filtre de périmètre de scan (US15, #31) : `{ include: Vec<String>, exclude: Vec<String>, respect_gitignore: bool }`. Porte les **motifs bruts validés**, jamais un matcher compilé — la **compilation des globs vit dans l'adaptateur** (`globset`), l'hexagone restant zéro-dep ([[ADR-0001]], [[ADR-0019]] D1). Auto-valide (`new` rejette motif vide/NUL/absolu/`..`/> 512 car/> 256 motifs → `FileFilterError`) : path-traversal et glob-DoS **inconstructibles**. `unrestricted()` = aucun filtrage = walk pré-US15. Règle de garde : `exclude` **l'emporte sur** `include` ; `include` vide = tout accepté. Voir [[ADR-0019]]. |
| include / exclude (globs) | Concept | Les deux listes de globs de `FileFilter` (matchées sur le chemin **relatif à la racine** du walk, moteur crate `ignore` + `globset`). Un fichier est retenu ssi son extension est demandée **ET** (`include` vide **OU** matché par `include`) **ET** non matché par `exclude` — **`exclude` gagne** sur `include`. Motifs validés à la frontière du VO `FileFilter` (voir [[ADR-0019]]). |
| respectGitignore | Concept | Booléen de `.codeimpact.json` (US15) : quand `true`, le walk honore **les quatre** sources d'ignore de la crate `ignore` — `.gitignore`, `.git/info/exclude`, le gitignore **global de la machine**, et `.ignore` — qui **gatent toutes ensemble** (ne gater qu'une source largue des fichiers en silence). Absent ⇒ défaut `true` (choisir un fichier de config est explicite) ; sans fichier ⇒ `unrestricted` = aucune source honorée (byte-identique au pré-US15). `.parents(false)` borne le walk à la racine analysée (pas de remontée aux `.gitignore` d'ancêtres). Résiduel `git_global` sur hôte CI mutualisé : documenté et accepté ([[ADR-0019]] frontière de confiance, [[ADR-0006]]). Voir [[ADR-0019]]. |
| Language | VO | Le langage d'un fichier source comme **type de première classe** de l'hexagone (zéro-dép) : enum `Rust` / `CSharp`, avec `from_extension` / `extensions` / `display_name`. Introduit par US14-T2 (#33). Voir [[ADR-0020]]. |
| ParserRegistry | Domain Service | Dispatche `Language -> Box<dyn CodeParser>` **par extension de fichier**, peuplé à la **racine de composition** (`.rs` → `SynCodeParser`, `.cs` → `TreeSitterCodeParser`). Un fichier d'extension inconnue est **sauté proprement** — jamais parsé-comme-du-Rust, jamais un panic. Deux moteurs derrière **un** port `CodeParser` : la preuve que le port agnostique d'[[ADR-0018]] est correct. Voir [[ADR-0020]]. |
| TreeSitterCodeParser | Adapter (secondaries) | **Un seul** adaptateur générique, paramétré par un `LanguageProfile` — **pas** un adaptateur par langage. Parse via `tree-sitter`, extrait les métriques par une requête `.scm` puis les plie dans le `ParsedFunction` neutre par un post-processeur générique **itératif** (zéro récursion). C# derrière la feature Cargo `lang-csharp`. Isolé du parseur natif par des **gardes in-process** (tree-sitter est table-driven, pas de crash process-level — voir [[ADR-0020]] § D5, amendement d'[[ADR-0015]]). Voir [[ADR-0020]]. |
| LanguageProfile | VO (secondaries) | Les **trois seules** données qui changent d'un langage à l'autre pour `TreeSitterCodeParser` : la **grammaire** tree-sitter, la **requête `.scm`** (capture fonctions + points de décision), et la **table de signatures d'I/O** ([[ADR-0016]]). Ajouter un langage (ex. TypeScript, #34) = ajouter un `LanguageProfile`, **pas** un adaptateur : le comportement par-langage est de la **donnée, pas du code**. Voir [[ADR-0020]]. |
| LanguageCapabilities / MetricSupport | VO / Enum | Couture de **dégradation par-métrique par-langage**, déclarée par le port `CodeParser::capabilities()`. `LanguageCapabilities` mappe `métrique -> MetricSupport` où `MetricSupport` ∈ `Supported` / `Degraded(reason)` / `Unsupported`. Posée (inerte) en T2 ([[ADR-0020]]) ; **allumée en T3** ([[ADR-0021]], #33) : la capacité **circule** via `CodeMetrics.capabilities` jusqu'aux **trois writers**, qui branchent **sur `MetricSupport`, jamais sur le langage** → une métrique non-`Supported` est rendue `n/a`, jamais `0`. C# à la sortie de T3 : cyclomatique/éco/écolo `Supported`, `call_graph` `Degraded("name-based resolution; ambiguous edges dropped")`, `io_in_loops` `Unsupported` (jusqu'à T4). Voir [[ADR-0021]], [[ADR-0020]]. |
| Dégradation honnête | Principe | Quand un langage ne sait pas produire une métrique, le rapport dit **`n/a`, jamais `0`** (un `0` se lit « gratuit / code propre » — un mensonge). Extension au *support d'un langage* du principe d'[[ADR-0010]] (« `Unmeasurable` plutôt que `0` », pour la *mesure* d'un run). Rendue de bout en bout par le canal `MetricSupport` → `CodeMetrics.capabilities` → writers. Le branchement **par état, non par langage** fait composer T4 (`io_in_loops` : `Unsupported`→`Degraded`) et T5 sans rouvrir les writers. Voir [[ADR-0021]], [[ADR-0010]]. |
| n/a (rendu de dégradation) | Concept | La forme d'affichage d'une métrique `Degraded`/`Unsupported`, par format : **Console** → `n/a — non supporté pour C#` (fr) + note `[dégradé: reason]` ; **JSON** → le champ sérialise `null` (jamais `[]` ni `0`) + un objet `metric_support` (valeurs **anglaises** `"supported"`/`"degraded: reason"`/`"unsupported"`, [[ADR-0007]] §7.9) ; **HTML** → `MetricVm.support`/`note`, `support` mappé par une whitelist JS **fermée** `SUP` vers les classes CSS `sup-ok`/`sup-degraded`/`sup-na` (pas d'`innerHTML`, discipline [[ADR-0008]] §8.10/§8.12). Voir [[ADR-0021]]. |
| Comptage cyclomatique inter-langages | Concept | `cyclomatique = 1 + Σ points de décision`, **épinglé sur `syn`** pour que le même motif structurel donne le même nombre quel que soit le langage (sinon les seuils d'[[ADR-0017]] perdent leur sens). Extension délibérée (signée humaine) : le **ternaire C# (`?:`) compte +1** — Rust n'a pas de ternaire, ce n'est donc pas une divergence mais la couverture d'une construction que `syn` n'avait jamais à classer. Voir [[ADR-0020]]. |
