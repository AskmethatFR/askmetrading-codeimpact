## Overview

The HTML report is the **human-shareable** output of `CodeImpact` analysis: a
single self-contained `.html` file (inline CSS + JS + JSON data island, system
fonts, zero external asset) that opens from `file://`. Produced by
`HtmlReportWriter` in `secondaries`. Contrast with the machine-readable JSON
([[json-report-schema]]) and the terminal-oriented console report
([[console-report-enriched]]).

Generated by:
```
codeimpact analyze --path <dir> --format html -o report.html
```
`--format html` requires a **project** (directory) target; a single-file target
errors (project view only — node-detail is staged T2, below).

## Architecture

**Port** (`hexagon/src/analysis/report_writer.rs`):
```rust
fn write_html(&self, graph: &FileConsumptionGraph, target: &str)
    -> Result<String, AnalysisError>;
```
Returns the full HTML document as a `String` — the hexagon never touches the
filesystem (ADR-0001 dependency rule). The file write lives in
`primaries/src/main.rs` (`write_html_report`, ADR-0006 discipline: canonicalize
parent dir, path-anonymised errors). Default output path when `-o` omitted:
`./report.html`.

`OutputFormat::Html` (pure enum, hexagon) + `handle_project_html` on
`RunAnalysis` select the writer, mirroring the JSON path (ADR-0007). Both
`handle_project_json` and `handle_project_html` share `build_project_graph`
(extracted to remove duplication).

The other three `ReportWriter` implementors (`ConsoleReportWriter`,
`JsonReportWriter`, `SharedReportWriterStub`) implement `write_html`:
console/json return `Err` (only `HtmlReportWriter` renders HTML); the stub
captures `last_html` for tests.

## View-model (presentation DTOs, secondaries only)

`#[derive(Serialize)]` DTOs beside the adapter — never on hexagon types (ADR-8.4):

```rust
struct ReportVm  { project: ProjectVm, files: Vec<FileNodeVm> }
struct ProjectVm { target: String, file_count: usize }
struct FileNodeVm {
    path: String,
    kind_label: String,   // T1: constant "FILE" (no per-file kind on the graph yet)
    score: u32,           // = transitive_complexity() — display heuristic (AD-4)
    score_pct: u8,        // score normalised to project max (bar width); 0 when max_score == 0
    level_label: String,  // = complexity_level()
}
```

`build_report_vm(graph, target)` iterates `graph.per_file_metrics()`, computes
the project max score for bar normalisation, and sorts files by path. Score is
`transitive_complexity()` — **a display heuristic, not a new domain metric**
(ADR-8.8): inventing a scoring concept would leak business logic into the
adapter (ca-layering).

## Document structure & rendering

`render_html(vm)` emits:
```html
<!DOCTYPE html><html lang="en"><head>… <style>{CSS}</style></head>
<body>
  <main id="ci-root"></main>
  <script id="ci-data" type="application/json">{escaped-VM-JSON}</script>
  <script>{JS}</script>
</body></html>
```
- `CSS`, `JS` are static Rust `const` string literals (no runtime template engine, ADR-8.3).
- The data island = `serde_json::to_string(vm)` passed through `json_island_escape`.
- The inline JS `JSON.parse`s the island and builds the DOM: every code-derived
  value (`file.path`, `file.kind_label`, `file.level_label`, `data.project.target`)
  is written via `textContent`/`createElement` — **never `innerHTML`**.

## Security — XSS defense (the load-bearing decision)

Two layers (ADR-8.5 / ADR-8.6):

1. **`textContent`/`createElement` rendering** — DOM text nodes are never
   HTML-parsed, so injection from a malicious file/function name is
   **structurally impossible** for dynamic content. This is the primary defense,
   not the escaping.
2. **`json_island_escape`** — closes the one residual vector, a literal
   `</script>` breaking out of the island (browsers scan raw `<script>` bodies
   for `</script` regardless of `type`):
   ```
   '<' → <   '>' → >   '&' → &
   U+2028 →     U+2029 →     (defense in depth)
   ```

CSP is not applicable to a `file://` document — the two layers above are the
whole defense, and were judged sufficient (Security: 0 critical/high/medium).
Verified live end-to-end with a real file named with an `<img onerror>` payload
(neutralised as text), plus an independent mutation check: bypassing
`json_island_escape` fails exactly the 2 breakout integration tests, proving the
tests bite.

**Accepted risk (LOW, A01/CWE-22)**: `-o ../x.html` writes outside cwd. No trust
boundary crossed (local single-user CLI). Documented, not fixed (ADR-8.9).
